# Google Play Store Launch Plan

Current status:

- The local runnable app is the `client` + `server` PWA/Express app.
- The app has PWA metadata at `client/public/manifest.json`.
- Standard `192x192` and `512x512` PNG icons are now available for PWA/TWA tooling.
- The service worker now caches Kharcha assets instead of the old Quantis icon.
- Authenticated account deletion is now available at `DELETE /api/auth/account` and in the Settings screen.
- A public account-deletion information page is available at `/account-deletion.html`.

## Recommended Packaging Path

Use a Trusted Web Activity (TWA) with Bubblewrap. This is the lowest-risk route
because Kharcha is already a mobile-first PWA and the backend must remain online
for account, grocery, pantry, finance, mandi, meal, and assistant features.

The Android app bundle will point to the production HTTPS web app URL.

## Blocking Inputs

Before generating the Play-ready Android App Bundle, decide these values:

- Production HTTPS URL, for example `https://kharcha.example.com`
- Android package name, for example `in.yourcompany.kharcha`
- Google Play developer account owner
- App signing approach in Play Console
- Developer/contact email
- Privacy policy URL
- Account deletion URL, usually `https://your-domain/account-deletion.html`
- Store screenshots and feature graphic

## Deploy Web App First

The TWA needs a stable public HTTPS URL. For the current working app, deploy the
single Express service that serves both `/api/*` and the built Vite app.

The included `render.yaml` deploys one Render web service backed by Render
Postgres. Render will provide a URL like:

```text
https://kharcha.onrender.com
```

To create that URL:

1. Commit and push this repo to GitHub.
2. Open Render and choose **New > Blueprint**.
3. Connect the GitHub repository.
4. Use the default Blueprint file path: `render.yaml`.
5. Apply the Blueprint.
6. Wait for the first deploy to finish.
7. Open `https://your-render-service.onrender.com/api/health`.
8. Open `https://your-render-service.onrender.com/manifest.json`.

Those two URLs must return valid responses before running Bubblewrap.

Equivalent production commands:

```bash
npm ci --include=dev
npm ci --prefix client
npm run build --prefix client
NODE_ENV=production PORT=3001 DATABASE_URL=... node apps/server/index.js
```

Production environment variables:

```bash
NODE_ENV=production
SESSION_EXPIRY_DAYS=30
DEV_OTP_VISIBLE=false
DATABASE_URL=...
ANTHROPIC_API_KEY=...
```

For the demo Render blueprint, `DEV_OTP_VISIBLE` is set to `true` because there
is no SMS provider integrated yet. Before a real Play Store production release,
integrate SMS OTP delivery and set `DEV_OTP_VISIBLE=false`.

Render's Free web service tier is acceptable for a first smoke test, but it is
not a production tier. Upgrade before public Play Store release to avoid slow
starts and free-tier limitations.

## Generate TWA Android Project

After the production URL is live:

```bash
npm install -g @bubblewrap/cli
bubblewrap init --manifest=https://your-domain/manifest.json
bubblewrap build --aab
```

Before upload, confirm the generated Android project targets the current Google
Play requirement, Android 15 / API level 35 or newer.

## Digital Asset Links

TWA verification requires a Digital Asset Links file on the web app:

```text
https://your-domain/.well-known/assetlinks.json
```

Use the SHA-256 fingerprint for the certificate that signs the Play-distributed
app. If Play App Signing is enabled, use the app signing certificate fingerprint
from Play Console, not only the local upload key.

## Play Console Checklist

- Create the app in Play Console.
- Upload the signed `.aab`.
- Fill Store listing: app name, short description, full description, category,
  screenshots, feature graphic, contact details.
- Fill App content: Privacy policy, Data safety, Ads, App access, Target
  audience, Content rating, News declaration if applicable.
- Provide test login instructions for reviewers. With production OTP delivery,
  explain exactly how reviewers can receive or use a test OTP.
- Release to internal testing first, then closed/open testing, then production.

## Data Safety Notes

Kharcha can collect or store:

- Phone number for OTP login.
- Profile and household preferences.
- Grocery lists, pantry items, meal plans, and chat messages.
- Finance records such as transactions, bills, subscriptions, and budgets.
- Technical logs generated by hosting infrastructure.

If the assistant feature is enabled with Anthropic, disclose that user chat
messages may be processed by that AI provider to generate responses.

Have the final privacy policy reviewed before publishing. The Play Console
declarations must match the app behavior and production vendor setup.
